1. EXECUTIVE SUMMARY
================================================================================

PROJECT PURPOSE:
The Ecom.Reporting microservice is a dedicated reporting and auditing system 
within an e-commerce platform. It tracks all domain events (product approvals, 
price changes, inventory updates, etc.) and provides dashboard summaries and 
exportable reports for administrators.

KEY FEATURES:
✓ Audit logging for all domain events
✓ Dashboard with real-time summaries
✓ Report export (CSV format)
✓ Event-driven architecture (RabbitMQ consumers)
✓ JWT-based authentication & authorization
✓ Health checks for database and messaging
✓ Clean Architecture with clear separation of concerns

TECHNOLOGY STACK:
- .NET 8.0 (C#)
- Entity Framework Core 8.0 (Code-First)
- SQL Server
- RabbitMQ (Event Messaging)
- AutoMapper (Object Mapping)
- FluentValidation (Input Validation)
- JWT Bearer Authentication
- Swagger/OpenAPI
- NUnit + Moq (Testing)
- Docker (Containerization)

3. CLEAN ARCHITECTURE BREAKDOWN
================================================================================

WHAT IS CLEAN ARCHITECTURE?
Clean Architecture (also called Onion Architecture or Hexagonal Architecture) 
is a software design pattern that separates concerns into layers, with 
dependencies pointing inward toward the core domain.

DEPENDENCY RULE:
Inner layers should NOT depend on outer layers.
Outer layers CAN depend on inner layers.

LAYER HIERARCHY (Inside → Outside):

┌─────────────────────────────────────────────────────────────┐
│                    PRESENTATION LAYER                        │
│              (Ecom.Reporting.API)                           │
│  • Controllers                                               │
│  • Middleware                                                │
│  • Program.cs (Dependency Injection Setup)                  │
└──────────────────────┬──────────────────────────────────────┘
                       │ Depends on ↓
┌─────────────────────────────────────────────────────────────┐
│                   APPLICATION LAYER                          │
│           (Ecom.Reporting.Application)                      │
│  • Business Logic (Services)                                │
│  • DTOs (Data Transfer Objects)                             │
│  • Interfaces (Abstractions)                                │
│  • Validators                                                │
│  • Mappings                                                  │
└──────────────────────┬──────────────────────────────────────┘
                       │ Depends on ↓
┌─────────────────────────────────────────────────────────────┐
│                     DOMAIN LAYER                             │
│              (Ecom.Reporting.Domain)                        │
│  • Entities (Core Business Models)                          │
│  • Enums                                                     │
│  • Value Objects                                             │
│  • Domain Rules                                              │
│  ⚠️ NO DEPENDENCIES ON OTHER LAYERS                         │
└─────────────────────────────────────────────────────────────┘
                       ↑ Depended on by
┌─────────────────────────────────────────────────────────────┐
│                 INFRASTRUCTURE LAYER                         │
│           (Ecom.Reporting.Infrastructure)                   │
│  • Database Context (EF Core)                               │
│  • Repository Implementations                                │
│  • External Service Integrations (RabbitMQ)                 │
│  • Migrations                                                │
└─────────────────────────────────────────────────────────────┘

WHY EACH LAYER EXISTS:

1. DOMAIN LAYER (Core):
   Purpose: Contains pure business logic and entities
   Why: Represents the heart of the application - business rules that never change
   Dependencies: NONE (completely independent)
   Example: AuditLog entity, AuditAction enum

2. APPLICATION LAYER:
   Purpose: Orchestrates business workflows and use cases
   Why: Coordinates between domain and infrastructure without knowing implementation details
   Dependencies: Domain Layer only
   Example: AuditService coordinates audit logging workflow

3. INFRASTRUCTURE LAYER:
   Purpose: Implements technical details (database, messaging, external APIs)
   Why: Keeps technical implementation separate from business logic
   Dependencies: Domain + Application (implements interfaces defined in Application)
   Example: AuditRepository implements IAuditRepository

4. PRESENTATION LAYER (API):
   Purpose: Exposes functionality to external consumers
   Why: Provides HTTP endpoints for clients to interact with the system
   Dependencies: Application + Infrastructure (for DI registration)
   Example: AuditController exposes audit endpoints

BENEFITS OF THIS ARCHITECTURE:
✓ Testability: Easy to mock dependencies and test in isolation
✓ Maintainability: Changes in one layer don't affect others
✓ Flexibility: Can swap implementations (e.g., SQL Server → PostgreSQL)
✓ Scalability: Clear boundaries make it easier to scale specific layers
✓ Domain-Centric: Business logic is protected and independent

LINE-BY-LINE EXPLANATION:

Line 1: namespace Ecom.Reporting.Domain.Entities;
→ Defines the namespace for domain entities

Line 3: public class AuditLog
→ Defines the AuditLog entity (represents a single audit trail record)

Line 5: public int Id { get; set; }
→ Primary Key (auto-generated by database)
→ Uniquely identifies each audit log entry

Line 6: public string? EntityName { get; set; }
→ Name of the entity being audited (e.g., "Product", "ProductVariant")
→ Nullable (?) because some events might not relate to a specific entity
→ Business Meaning: Tracks WHAT type of object was affected

Line 7: public int? EntityId { get; set; }
→ ID of the specific entity instance (e.g., ProductId = 123)
→ Nullable because some events are system-wide
→ Business Meaning: Tracks WHICH specific object was affected

Line 8: public string? Action { get; set; }
→ The action performed (e.g., "Created", "Updated", "Approved", "Published")
→ Business Meaning: Tracks WHAT HAPPENED to the entity

Line 9: public string? EventType { get; set; }
→ The domain event type (e.g., "product.approved", "inventory.updated")
→ Business Meaning: Technical event name from the messaging system

Line 10: public string? SourceService { get; set; }
→ Which microservice triggered this event (e.g., "WorkflowService")
→ Business Meaning: Tracks WHERE the event originated (important in microservices)

Line 11: public string? CorrelationId { get; set; }
→ Unique identifier to trace a request across multiple services
→ Business Meaning: Enables distributed tracing in microservice architecture

Line 12: public DateTime CreatedAt { get; set; } = DateTime.UtcNow;
→ Timestamp when the audit log was created
→ Default value: Current UTC time
→ Business Meaning: Tracks WHEN the event occurred

AGGREGATE ROOT:
✓ AuditLog is an Aggregate Root (it's a standalone entity with its own lifecycle)
✓ No child entities or value objects in this simple domain

DOMAIN RULES:
1. CreatedAt is always set to UTC time (prevents timezone issues)
2. All string properties are nullable (flexible for different event types)
3. Id is the only required field (enforced by database)

SERVICE 1: AuditService.cs
```csharp
using AutoMapper;
using Ecom.Reporting.Application.DTOs;
using Ecom.Reporting.Application.Interfaces;
using Ecom.Reporting.Domain.Entities;

namespace Ecom.Reporting.Application.Services;

public class AuditService : IAuditService
{
    private readonly IAuditRepository _repo;
    private readonly IMapper _mapper;

    public AuditService(IAuditRepository repo, IMapper mapper)
    {
        _repo = repo;
        _mapper = mapper;
    }

    public async Task<IEnumerable<AuditLogDto>> GetByProductIdAsync(int productId)
    {
        var logs = await _repo.GetByEntityAsync("Product", productId);
        return _mapper.Map<IEnumerable<AuditLogDto>>(logs);
    }

    public async Task<IEnumerable<AuditLogDto>> GetAllAsync(DateTime? from, DateTime? to)
    {
        var logs = await _repo.GetAllAsync(from, to);
        return _mapper.Map<IEnumerable<AuditLogDto>>(logs);
    }

    public async Task WriteAsync(AuditLogDto dto)
    {
        var entity = _mapper.Map<AuditLog>(dto);
        await _repo.AddAsync(entity);
    }
}
```

Lines 9-10: Dependencies
→ IAuditRepository: Data access abstraction
→ IMapper: AutoMapper for DTO ↔ Entity conversion

Lines 12-16: Constructor Injection
→ Dependencies injected via constructor (Dependency Injection pattern)
→ Stored in private readonly fields

Lines 18-22: GetByProductIdAsync
→ Calls repository with hardcoded "Product" entity name
→ Maps domain entities to DTOs using AutoMapper
→ Returns DTOs to caller (controller)

Lines 24-28: GetAllAsync
→ Retrieves all audit logs with optional date filtering
→ Maps to DTOs before returning

Lines 30-34: WriteAsync
→ Receives DTO from controller

→ Persists to database via repository

SERVICE 3: ReportExportService.cs
```csharp
using System.Text;
using Ecom.Reporting.Application.DTOs;
using Ecom.Reporting.Application.Interfaces;

namespace Ecom.Reporting.Application.Services;

public class ReportExportService : IReportExportService
{
    private readonly IAuditRepository _repository;

    public ReportExportService(IAuditRepository repository) => _repository = repository;

    public async Task<byte[]> ExportAsync(ReportExportRequestDto request)
    {
        if (string.IsNullOrWhiteSpace(request.ReportType))
            throw new InvalidOperationException("Report type is required.");

        var logs = await _repository.GetFilteredAsync(
            request.FromDate, 
            request.ToDate, 
            request.ReportType, 
            10000);

        var sb = new StringBuilder();
        sb.AppendLine("Id,EntityName,EntityId,Action,EventType,SourceService,CorrelationId,CreatedAt");

        foreach (var log in logs)
        {
            sb.AppendLine($"{log.Id},{log.EntityName},{log.EntityId},{log.Action},{log.EventType},{log.SourceService},{log.CorrelationId},{log.CreatedAt:O}");
        }

        return Encoding.UTF8.GetBytes(sb.ToString());
    }
}
```

LINE-BY-LINE EXPLANATION:

Lines 15-16: Validation
→ Ensures ReportType is provided
→ Throws InvalidOperationException if missing (caught by middleware)

Lines 18-22: Data Retrieval
→ Calls repository with filters
→ Limit: 10,000 records (prevents memory issues)

Lines 24-25: CSV Header
→ Creates CSV header row with column names

Lines 27-30: CSV Body
→ Iterates through logs and builds CSV rows
→ {log.CreatedAt:O}: ISO 8601 format (round-trip date/time pattern)

Line 32: Return Bytes
→ Converts string to UTF-8 byte array
→ Ready for HTTP file download

5.4 VALIDATORS

VALIDATOR: ReportExportRequestValidator.cs
```csharp
using Ecom.Reporting.Application.DTOs;
using FluentValidation;

namespace Ecom.Reporting.Application.Validators;

public class ReportExportRequestValidator : AbstractValidator<ReportExportRequestDto>
{
    private static readonly string[] ValidReportTypes = 
        { "catalog-quality", "low-stock", "price-changes", "all-audit" };

    public ReportExportRequestValidator()
    {
        RuleFor(x => x.ReportType)
            .NotEmpty()
            .WithMessage("Report type is required")
            .Must(type => ValidReportTypes.Contains(type.ToLower()))
            .WithMessage($"Invalid report type. Valid types: {string.Join(", ", ValidReportTypes)}");

        RuleFor(x => x.Format)
            .Must(f => f == "csv" || f == "excel")
            .WithMessage("Format must be 'csv' or 'excel'");

        RuleFor(x => x.FromDate)
            .LessThan(x => x.ToDate)
            .When(x => x.FromDate.HasValue && x.ToDate.HasValue)
            .WithMessage("FromDate must be before ToDate");
    }
}
```

EXPLANATION:
→ FluentValidation library for declarative validation
→ Validates ReportExportRequestDto before processing
→ Rules:
  1. ReportType must be one of the valid types
  2. Format must be "csv" or "excel"
  3. FromDate must be before ToDate (if both provided)

```csharp
using Ecom.Reporting.Domain.Entities;
```

```csharp
namespace Ecom.Reporting.Infrastructure.Persistence;

public class ReportingDbContext : DbContext
{
    public ReportingDbContext(DbContextOptions<ReportingDbContext> options)

    public DbSet<AuditLog> AuditLogs => Set<AuditLog>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
```

```csharp
using Ecom.Reporting.Application.Interfaces;
using Ecom.Reporting.Domain.Entities;
using Ecom.Reporting.Infrastructure.Persistence;
using Microsoft.EntityFrameworkCore;

namespace Ecom.Reporting.Infrastructure.Repositories;

public class AuditRepository : IAuditRepository
{
    private readonly ReportingDbContext _context;

    public AuditRepository(ReportingDbContext context) => _context = context;

    public async Task<IEnumerable<AuditLog>> GetByEntityAsync(string entityName, int entityId) =>
        await _context.AuditLogs
            .Where(a => a.EntityName == entityName && a.EntityId == entityId)
            .OrderByDescending(a => a.CreatedAt)
            .ToListAsync();

    public async Task<IEnumerable<AuditLog>> GetAllAsync(DateTime? from, DateTime? to)
    {
        var query = _context.AuditLogs.AsQueryable();
        if (from.HasValue) query = query.Where(a => a.CreatedAt >= from.Value);
        if (to.HasValue) query = query.Where(a => a.CreatedAt <= to.Value);
        return await query.OrderByDescending(a => a.CreatedAt).ToListAsync();
    }

    public async Task AddAsync(AuditLog log)
    {
        await _context.AuditLogs.AddAsync(log);
        await _context.SaveChangesAsync();
    }

    public async Task<Dictionary<string, int>> GetActionCountsAsync()
    {
        return await _context.AuditLogs
            .GroupBy(a => a.Action)
            .Select(g => new { Action = g.Key, Count = g.Count() })
            .ToDictionaryAsync(x => x.Action ?? "Unknown", x => x.Count);
    }

    public async Task<IEnumerable<AuditLog>> GetFilteredAsync(
        DateTime? from, DateTime? to, string? reportType, int limit)
    {
        var query = _context.AuditLogs.AsQueryable();

        if (from.HasValue)
            query = query.Where(a => a.CreatedAt >= from.Value);
        if (to.HasValue)
            query = query.Where(a => a.CreatedAt <= to.Value);

        if (!string.IsNullOrWhiteSpace(reportType))
        {
            query = reportType.ToLower() switch
            {
                "catalog-quality" => query.Where(a => a.EntityName == "Product"),
                "low-stock" => query.Where(a => a.Action == "StockChanged"),
                "price-changes" => query.Where(a => a.Action == "PriceChanged"),
                "all-audit" => query,
                _ => throw new InvalidOperationException(
                    $"Unknown report type: {reportType}")
            };
        }

        return await query
            .OrderByDescending(a => a.CreatedAt)
            .Take(limit)
            .ToListAsync();
    }
}
```

FILE: RabbitMqConsumerBase.cs (Base Class)
```csharp
using System.Text;
using Microsoft.Extensions.Configuration;
using Microsoft.Extensions.DependencyInjection;
using Microsoft.Extensions.Hosting;
using Microsoft.Extensions.Logging;
using RabbitMQ.Client;
using RabbitMQ.Client.Events;

namespace Ecom.Reporting.Infrastructure.Messaging;

public abstract class RabbitMqConsumerBase : BackgroundService
{
    protected IConnection? _connection;
    protected IModel? _channel;
    protected readonly IConfiguration _configuration;
    protected readonly IServiceProvider _serviceProvider;
    protected readonly ILogger _logger;
    
    protected abstract string QueueName { get; }
    protected abstract string RoutingKey { get; }

    protected RabbitMqConsumerBase(
        IConfiguration configuration,
        IServiceProvider serviceProvider,
        ILogger logger)
    {
        _configuration = configuration;
        _serviceProvider = serviceProvider;
        _logger = logger;
    }

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        try
        {
            InitializeRabbitMQ();
            
            if (_channel == null || !_channel.IsOpen)
            {
                _logger.LogWarning("RabbitMQ channel not available for {Queue}", QueueName);
                return;
            }

            var consumer = new EventingBasicConsumer(_channel);
            consumer.Received += async (model, ea) =>
            {
                try
                {
                    var body = ea.Body.ToArray();
                    var message = Encoding.UTF8.GetString(body);
                    
                    _logger.LogInformation("Received message on {Queue}: {Message}", 
                        QueueName, message);
                    
                    await ProcessMessageAsync(message);
                    
                    _channel.BasicAck(ea.DeliveryTag, false);
                }
                catch (Exception ex)
                {
                    _logger.LogError(ex, "Error processing message on {Queue}", QueueName);
                    _channel.BasicNack(ea.DeliveryTag, false, true);
                }
            };

            _channel.BasicConsume(queue: QueueName, autoAck: false, consumer: consumer);
            
            _logger.LogInformation("Consumer started for {Queue}", QueueName);
            
            await Task.Delay(Timeout.Infinite, stoppingToken);
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Fatal error in consumer {Queue}", QueueName);
        }
    }

    private void InitializeRabbitMQ()
    {
        try
        {
            var factory = new ConnectionFactory
            {
                HostName = _configuration["RabbitMq:Host"] ?? "localhost",
                Port = int.Parse(_configuration["RabbitMq:Port"] ?? "5672"),
                UserName = _configuration["RabbitMq:Username"] ?? "guest",
                Password = _configuration["RabbitMq:Password"] ?? "guest"
            };

            _connection = factory.CreateConnection();
            _channel = _connection.CreateModel();

            _channel.ExchangeDeclare(
                exchange: "ecom.domain.events",
                type: ExchangeType.Topic,
                durable: true,
                autoDelete: false);

            _channel.QueueDeclare(
                queue: QueueName,
                durable: true,
                exclusive: false,
                autoDelete: false);

            _channel.QueueBind(
                queue: QueueName,
                exchange: "ecom.domain.events",
                routingKey: RoutingKey);

            _logger.LogInformation("RabbitMQ initialized for {Queue} with routing key {RoutingKey}", 
                QueueName, RoutingKey);
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Failed to initialize RabbitMQ for {Queue}", QueueName);
        }
    }

    protected abstract Task ProcessMessageAsync(string message);

    public override void Dispose()
    {
        _channel?.Close();
        _connection?.Close();
        base.Dispose();
    }
}
```

LINE-BY-LINE EXPLANATION:

Line 11: BackgroundService
→ Base class for long-running background tasks
→ Runs continuously in the background

Lines 19-20: Abstract Properties
→ Derived classes must provide QueueName and RoutingKey
→ Template Method Pattern

Lines 32-74: ExecuteAsync (Main Loop)
→ Called when the background service starts
→ Initializes RabbitMQ connection
→ Sets up message consumer
→ Runs indefinitely until cancellation

Lines 44-64: Message Handler
→ Received event: Triggered when message arrives
→ Deserializes message body
→ Calls ProcessMessageAsync (implemented by derived class)
→ BasicAck: Acknowledges successful processing
→ BasicNack: Rejects message on error (requeue = true)

Lines 76-115: InitializeRabbitMQ
→ Creates connection to RabbitMQ server
→ Declares exchange (ecom.domain.events)
→ Declares queue (specific to consumer)
→ Binds queue to exchange with routing key

RABBITMQ CONCEPTS:
- Exchange: Routes messages to queues based on routing keys
- Queue: Stores messages until consumed
- Routing Key: Pattern for message routing (e.g., "product.approved")
- Topic Exchange: Routes based on pattern matching

```csharp
public class ProductApprovedConsumer : RabbitMqConsumerBase
{
    protected override string QueueName => "reporting.product.approved";
    protected override string RoutingKey => "product.approved";

    public ProductApprovedConsumer(
        IConfiguration configuration,
        IServiceProvider serviceProvider,
        ILogger<ProductApprovedConsumer> logger)
        : base(configuration, serviceProvider, logger)
    {
    }

    protected override async Task ProcessMessageAsync(string message)
    {
        using var scope = _serviceProvider.CreateScope();
        var auditService = scope.ServiceProvider.GetRequiredService<IAuditService>();

        var eventData = JsonSerializer.Deserialize<ProductApprovedEvent>(message);
        
        if (eventData == null)
        {
            _logger.LogWarning("Failed to deserialize product.approved event");
            return;
        }

        await auditService.WriteAsync(new AuditLogDto
        {
            EntityName = "Product",
            EntityId = eventData.ProductId,
            Action = "Approved",
            EventType = "product.approved",
            SourceService = "WorkflowService",
            CorrelationId = eventData.CorrelationId ?? Guid.NewGuid().ToString(),
            CreatedAt = DateTime.UtcNow
        });

        _logger.LogInformation("Processed product.approved for ProductId: {ProductId}", 
            eventData.ProductId);
    }

    private class ProductApprovedEvent
    {
        public int ProductId { get; set; }
        public int ApprovedBy { get; set; }
        public string? CorrelationId { get; set; }
    }
}
```

```csharp
public class ExceptionHandlingMiddleware

            context.Response.StatusCode = 500;
            await context.Response.WriteAsync(
                JsonSerializer.Serialize(new { error = "Internal server error." }));
        }
    }
}
```

8. DATABASE DESIGN - CODE-FIRST APPROACH ⭐⭐⭐
================================================================================

This is the MOST IMPORTANT section for understanding how the database is 
designed using Entity Framework Core's Code-First approach.

----------------------------
8.1 WHAT IS CODE-FIRST?
----------------------------

CODE-FIRST APPROACH:
Instead of designing the database first and then generating code, you:
1. Write C# entity classes (POCOs - Plain Old CLR Objects)
2. Configure relationships and constraints using Fluent API or Data Annotations
3. Generate database schema from code using Migrations
4. Apply migrations to create/update the database

BENEFITS:
✓ Version control for database schema (migrations are code)
✓ Database-agnostic (can switch from SQL Server to PostgreSQL)
✓ Type-safe (compile-time checking)
✓ Easier to refactor (rename properties, IDE helps)
✓ Better for team collaboration (merge conflicts are easier)

WORKFLOW:
Code (Entities) → EF Core → Migration Files → Database Schema

----------------------------
8.2 DBCONTEXT ANALYSIS
----------------------------

The DbContext is the bridge between your code and the database.

```csharp
public class ReportingDbContext : DbContext
{
    public ReportingDbContext(DbContextOptions<ReportingDbContext> options)

    public DbSet<AuditLog> AuditLogs => Set<AuditLog>();
```

8.6 FLUENT API CONFIGURATION

FILE: AuditLogConfiguration.cs

```csharp
public class AuditLogConfiguration : IEntityTypeConfiguration<AuditLog>

    public void Configure(EntityTypeBuilder<AuditLog> builder)

```

FLUENT API METHODS EXPLAINED:

1. HasKey(a => a.Id)
   → Defines primary key
   → SQL: PRIMARY KEY (Id)
   → Auto-generates IDENTITY(1,1) for int type

2. Property(a => a.EntityName).HasMaxLength(100)
   → Sets maximum string length
   → SQL: nvarchar(100)
   → Prevents unlimited string storage

3. HasIndex(a => new { a.EntityName, a.EntityId })
   → Creates composite index
   → SQL: CREATE INDEX IX_Audit_Entity ON AuditLogs (EntityName, EntityId)
   → Optimizes queries: WHERE EntityName = 'Product' AND EntityId = 123

4. HasDatabaseName("IX_Audit_Entity")
   → Custom index name (instead of auto-generated)
   → Makes index name predictable

OTHER FLUENT API METHODS (not used here):
- IsRequired(): Makes column NOT NULL
- HasDefaultValue(): Sets SQL default value
- HasColumnName(): Custom column name
- HasColumnType(): Custom SQL type
- HasOne() / WithMany(): Defines relationships
- HasForeignKey(): Defines foreign key
- OnDelete(): Cascade delete behavior

----------------------------
8.7 INDEXES EXPLAINED
----------------------------

INDEX: IX_Audit_Entity
Type: Non-Clustered Composite Index
Columns: (EntityName, EntityId)

WHY THIS INDEX?
Common query pattern:
```csharp
_context.AuditLogs
    .Where(a => a.EntityName == "Product" && a.EntityId == 123)
    .ToListAsync();
```

WITHOUT INDEX:
→ Full table scan (reads every row)
→ Slow for large tables (millions of rows)

WITH INDEX:
→ B-Tree lookup (logarithmic time)
→ Fast even for large tables

INDEX STRUCTURE (simplified):
```
EntityName | EntityId | Row Pointer
-----------|----------|------------
Product    | 1        | → Row 1
Product    | 2        | → Row 5
Product    | 3        | → Row 8
Variant    | 1        | → Row 2
Variant    | 2        | → Row 3
```

COMPOSITE INDEX ORDER MATTERS:
✓ Fast: WHERE EntityName = 'Product' AND EntityId = 123
✓ Fast: WHERE EntityName = 'Product'
✗ Slow: WHERE EntityId = 123 (can't use index efficiently)

8.8 DATA TYPES MAPPING
----------------------------

C# TYPE → SQL SERVER TYPE:

┌──────────────────┬──────────────────┬────────────────────────────┐
│ C# Type          │ SQL Server Type  │ Notes                      │
├──────────────────┼──────────────────┼────────────────────────────┤
│ int              │ int              │ 4 bytes, -2B to +2B        │
│ int?             │ int NULL         │ Nullable integer           │
│ string           │ nvarchar(MAX)    │ Unicode, unlimited         │
│ string (max 100) │ nvarchar(100)    │ Unicode, limited           │
│ DateTime         │ datetime2        │ High precision, 7 digits   │
│ bool             │ bit              │ True/False                 │
│ decimal          │ decimal(18,2)    │ Precise numbers (money)    │
│ Guid             │ uniqueidentifier │ 16-byte GUID               │
└──────────────────┴──────────────────┴────────────────────────────┘

WHY nvarchar INSTEAD OF varchar?
→ nvarchar: Unicode (supports all languages)
→ varchar: ASCII only (English, numbers, symbols)
→ E-commerce needs international support

WHY datetime2 INSTEAD OF datetime?
→ datetime2: Higher precision (100 nanoseconds)
→ datetime: Lower precision (3.33 milliseconds)
→ datetime2: Larger range (0001-01-01 to 9999-12-31)

----------------------------
11.5 ERROR HANDLING
----------------------------

ACKNOWLEDGMENT MODES:

1. SUCCESS (BasicAck):
```csharp
_channel.BasicAck(ea.DeliveryTag, false);
```
→ Message is removed from queue
→ Processing completed successfully

2. FAILURE (BasicNack with Requeue):
```csharp
_channel.BasicNack(ea.DeliveryTag, false, requeue: true);
```
→ Message is requeued
→ Will be redelivered
→ Allows retry on transient failures

3. FAILURE (BasicNack without Requeue):
```csharp
_channel.BasicNack(ea.DeliveryTag, false, requeue: false);
```
→ Message is discarded (or sent to dead-letter queue)
→ Used for poison messages (permanently invalid)

RETRY STRATEGY:
Current implementation: Infinite retries (requeue: true)

BETTER APPROACH (not implemented):
- Retry count tracking
- Exponential backoff
- Dead-letter queue for failed messages
- Manual intervention for poison messages

11.6 BENEFITS OF EVENT-DRIVEN ARCHITECTURE
----------------------------

✓ LOOSE COUPLING:
Services don't need to know about each other
Reporting service doesn't call Workflow service directly

✓ SCALABILITY:
Multiple consumers can process messages in parallel
Can add more consumer instances for high load

✓ RESILIENCE:
If Reporting service is down, messages are queued
No data loss (messages are persisted)

✓ ASYNCHRONOUS PROCESSING:
Workflow service doesn't wait for audit logging
Faster response times for users

✓ FLEXIBILITY:
Easy to add new consumers for new services
No changes to publishers

14.1 ENTITY: AuditLog

```csharp
namespace Ecom.Reporting.Domain.Entities;

public class AuditLog
{
    public int Id { get; set; }
    public string? EntityName { get; set; }
    public int? EntityId { get; set; }
    public string? Action { get; set; }
    public string? EventType { get; set; }
    public string? SourceService { get; set; }
    public string? CorrelationId { get; set; }
    public DateTime CreatedAt { get; set; } = DateTime.UtcNow;
}
```

WHY THIS CODE EXISTS:
→ Represents the core business entity for audit logging
→ POCO (Plain Old CLR Object) - no framework dependencies
→ Maps to AuditLogs table in database

DESIGN DECISIONS:
1. Nullable properties (string?, int?):
   → Flexibility for different event types
   → Some events may not have EntityId

2. DateTime.UtcNow default:
   → Ensures timestamp is always set
   → UTC prevents timezone issues

3. No navigation properties:
   → Standalone entity (no relationships)
   → Microservice architecture (entities in different databases)

WHAT PROBLEM IT SOLVES:
→ Provides audit trail for all system events
→ Enables compliance and debugging
→ Supports distributed tracing (CorrelationId)

// Add pagination to GetAllAsync
public async Task<PagedResult<AuditLog>> GetAllAsync(
    DateTime? from, DateTime? to, int page = 1, int pageSize = 50)
{
    var query = _context.AuditLogs.AsQueryable();
    
    if (from.HasValue) query = query.Where(a => a.CreatedAt >= from.Value);
    if (to.HasValue) query = query.Where(a => a.CreatedAt <= to.Value);

    var total = await query.CountAsync();
    var items = await query
        .OrderByDescending(a => a.CreatedAt)
        .Skip((page - 1) * pageSize)
        .Take(pageSize)
        .ToListAsync();

    return new PagedResult<AuditLog>
    {
        Items = items,
        TotalCount = total,
        Page = page,
        PageSize = pageSize
    };
}

    {

            ?? Guid.NewGuid().ToString();

        context.Response.Headers.Add("X-Correlation-ID", correlationId);

        using (_logger.BeginScope(new Dictionary<string, object>
        {

        }))
        {
            await _next(context);
        }
    }
}
16.5 DATA MANAGEMENT
DATA RETENTION POLICY
// Background service to archive old logs
public class DataRetentionService : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            try
            {
                var cutoffDate = DateTime.UtcNow.AddMonths(-6);
                
                // Archive logs older than 6 months
                var oldLogs = await _context.AuditLogs
                    .Where(a => a.CreatedAt < cutoffDate)
                    .Take(1000)
                    .ToListAsync(stoppingToken);

                if (oldLogs.Any())
                {
                    // Export to blob storage
                    await _blobService.ArchiveAsync(oldLogs);
                    
                    // Delete from database
                    _context.AuditLogs.RemoveRange(oldLogs);
                    await _context.SaveChangesAsync(stoppingToken);
                }

                await Task.Delay(TimeSpan.FromHours(24), stoppingToken);
            }
            catch (Exception ex)
            {
                _logger.LogError(ex, "Data retention job failed");
            }
        }
    }
}

# ADR-001: Use Clean Architecture

## Status
Accepted

## Context
Need to structure the codebase for maintainability and testability.

## Decision

================================================================================ 17. INTERVIEW PREPARATION GUIDE
This section prepares you for technical interviews about this project.

17.1 COMMON INTERVIEW QUESTIONS
Q1: "Explain the architecture of your reporting microservice."

ANSWER: "I implemented Clean Architecture with four distinct layers:

Domain Layer - Contains the core business entities like AuditLog and enums. It has zero dependencies on other layers or frameworks.

Application Layer - Contains business logic, DTOs, and service interfaces. It orchestrates workflows and depends only on the Domain layer.

Infrastructure Layer - Implements technical details like database access with Entity Framework Core and RabbitMQ messaging. It implements interfaces defined in the Application layer.

API Layer - Exposes HTTP endpoints using ASP.NET Core controllers. It depends on Application and Infrastructure layers.

The key benefit is the Dependency Inversion Principle - inner layers don't depend on outer layers, making the system testable and maintainable."

Q2: "How does your service handle database operations?"

ANSWER: "I used Entity Framework Core with a Code-First approach:

I defined the AuditLog entity as a POCO in the Domain layer
Created a DbContext (ReportingDbContext) in Infrastructure
Used Fluent API for entity configuration (AuditLogConfiguration)
Generated migrations to create the database schema
Implemented Repository Pattern for data access abstraction
The repository (AuditRepository) provides methods like GetByEntityAsync and AddAsync, which use LINQ queries that EF Core translates to parameterized SQL. This protects against SQL injection and provides type safety.

For performance, I added a composite index on (EntityName, EntityId) since that's a common query pattern."

Q3: "Explain your event-driven architecture."

ANSWER: "The service consumes domain events from RabbitMQ:

External services (Workflow, Inventory, Pricing) publish events to a topic exchange called 'ecom.domain.events'

I created four background consumers that inherit from RabbitMqConsumerBase:

ProductApprovedConsumer
ProductPublishedConsumer
InventoryUpdatedConsumer
PricingUpdatedConsumer
Each consumer listens to a specific queue bound to the exchange with a routing key (e.g., 'product.approved')

When a message arrives, the consumer deserializes it, creates an audit log entry via the AuditService, and acknowledges the message

If processing fails, the message is requeued for retry

This provides loose coupling between services, asynchronous processing, and fault tolerance since messages are persisted in RabbitMQ."

Q4: "How did you implement authentication and authorization?"

ANSWER: "I implemented JWT Bearer authentication:

Configured JWT authentication in Program.cs with token validation parameters
Specified issuer, audience, and signing key from configuration
Applied [Authorize(Roles = 'Admin')] attribute to controllers
Configured Swagger to support JWT tokens for testing
The authentication middleware validates the JWT token on each request, extracts claims, and populates the User principal. The authorization middleware then checks if the user has the required role.

For production, I'd recommend:

Using a dedicated identity service (IdentityServer, Auth0)
Implementing refresh tokens
Adding rate limiting
Using Azure Key Vault for secrets"
Q5: "What testing strategy did you use?"

ANSWER: "I followed the testing pyramid approach:

Unit Tests - Tested business logic in isolation using Moq for mocking dependencies. For example, AuditServiceTests mocks IAuditRepository and verifies the service behavior.

Integration Tests - Would test the full stack with a real database (in-memory or test container) to verify end-to-end functionality.

The key is testing behavior, not implementation. I use the Arrange-Act-Assert pattern and ensure tests are independent with no shared state.

For production, I'd add:

Load testing with tools like NBomber
Contract testing to ensure API compatibility
End-to-end tests for critical workflows"
Q6: "How would you optimize this service for high load?"

ANSWER: "Several optimization strategies:

Database Level:

Add indexes on frequently queried columns (CreatedAt, CorrelationId, Action)
Use AsNoTracking() for read-only queries (30-40% faster)
Implement pagination to limit result sets
Consider database partitioning by date for large tables
Application Level:

Add caching for dashboard data (5-minute cache)
Use async/await throughout (already implemented)
Implement connection pooling (configure in connection string)
Infrastructure Level:

Scale horizontally with multiple instances behind a load balancer
Use read replicas for reporting queries
Implement circuit breakers and retry policies with Polly
Messaging:

Add multiple consumer instances for parallel processing
Use prefetch count to control message batching
Implement dead letter queues for poison messages
Monitoring:

Add metrics (Prometheus) to identify bottlenecks
Use distributed tracing (OpenTelemetry) to track request flow
Set up alerts for performance degradation"
Q7: "Explain your database schema design."

ANSWER: "The database has a single table: AuditLogs

Columns:

Id (int, PK, IDENTITY) - Auto-incrementing primary key
EntityName (nvarchar(100)) - Type of entity (e.g., 'Product')
EntityId (int) - ID of the specific entity instance
Action (nvarchar(50)) - Action performed (e.g., 'Approved')
EventType (nvarchar(100)) - Technical event name
SourceService (nvarchar(50)) - Originating microservice
CorrelationId (nvarchar(100)) - For distributed tracing
CreatedAt (datetime2) - Timestamp in UTC
Indexes:

Clustered index on Id (primary key)
Non-clustered composite index on (EntityName, EntityId) for fast lookups
Design decisions:

No foreign keys because entities are in different databases (microservices)
Nullable columns for flexibility with different event types
nvarchar for Unicode support (international characters)
datetime2 for high precision timestamps
All string columns have max length for performance
For production, I'd add:

Index on CreatedAt for date range queries
Index on CorrelationId for tracing
Partitioning by date for large datasets
Archival strategy for old data"
