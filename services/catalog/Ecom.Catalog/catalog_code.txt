1.1 WHAT IS THE CATALOG SERVICE?
────────────────────────────────────
The Catalog Service is the PRODUCT MANAGEMENT BACKBONE of an e-commerce system.

CORE RESPONSIBILITIES:
• Product CRUD operations (Create, Read, Update, Delete)
• Category hierarchy management (nested categories)
• Brand management
• Product media/image management
• Product variants (size, color combinations)
• CQRS read model for high-performance queries
• Audit trail for compliance (GDPR, SOX)
• Event publishing for microservices communication

REAL-WORLD ANALOGY:
Think of it as the "Product Catalog Department" in a retail company:
- Manages product information
- Organizes products into categories
- Tracks product changes
- Provides product data to other departments (Order, Basket, Search)

SERVICE INTERACTIONS:

1. Identity Service → Catalog
   - Provides JWT tokens for authentication
   - Supplies UserId for CreatedBy field
   - Provides user email for audit logs

2. Catalog → Order Service
   - Order service reads product details
   - Validates product availability
   - Gets pricing information

3. Catalog → Basket Service
   - Basket validates products before adding
   - Checks product existence
   - Retrieves product names/images

4. Catalog → Workflow Service (via RabbitMQ)
   - Publishes ProductCreatedEvent
   - Publishes ProductPublishedEvent
   - Workflow updates ProductReadModel (CQRS)

5. Catalog → Search Service (not shown)
   - Publishes events for search index updates
   - Elasticsearch/Solr integration

1.3 WHY CLEAN ARCHITECTURE?
────────────────────────────

CLEAN ARCHITECTURE LAYERS (Onion Architecture):


        ┌─────────────────────────────────────────┐
        │         API Layer (Controllers)         │
        │  • HTTP endpoints                       │
        │  • Request/Response handling            │
        │  • Authentication/Authorization         │
        └──────────────┬──────────────────────────┘
                       │
        ┌──────────────▼──────────────────────────┐
        │    Application Layer (Services/DTOs)    │
        │  • Business workflows                   │
        │  • Validation                           │
        │  • Mapping (Entity ↔ DTO)              │
        └──────────────┬──────────────────────────┘
                       │
        ┌──────────────▼──────────────────────────┐
        │      Domain Layer (Entities/Rules)      │
        │  • Business entities                    │
        │  • Domain rules                         │
        │  • Domain events                        │
        │  • NO DEPENDENCIES (Pure C#)            │
        └──────────────┬──────────────────────────┘
                       │
        ┌──────────────▼──────────────────────────┐
        │  Infrastructure Layer (DB/External)     │
        │  • EF Core DbContext                    │
        │  • Repositories                         │
        │  • External services (RabbitMQ)         │
        └─────────────────────────────────────────┘

BENEFITS OF CLEAN ARCHITECTURE:

1. TESTABILITY
   - Domain layer has ZERO dependencies
   - Can test business logic without database
   - Mock interfaces easily

2. MAINTAINABILITY
   - Changes in one layer don't cascade
   - Clear separation of concerns
   - Easy to locate code

3. FLEXIBILITY
   - Swap SQL Server for PostgreSQL? Change Infrastructure only
   - Change API from REST to gRPC? Change API layer only
   - Domain logic remains untouched

4. SCALABILITY
   - Each layer can be optimized independently
   - Can split into separate microservices later
   - Clear boundaries

5. TEAM COLLABORATION
   - Frontend team works on API layer
   - Backend team works on Domain/Application
   - DBA team works on Infrastructure
   - No conflicts

REAL-WORLD EXAMPLE:
If you need to switch from SQL Server to MongoDB:
- Domain Layer: NO CHANGES (entities remain same)
- Application Layer: NO CHANGES (interfaces remain same)
- Infrastructure Layer: Rewrite repositories for MongoDB
- API Layer: NO CHANGES

3.1 PRODUCT ENTITY - COMPLETE CODE + EXPLANATION
─────────────────────────────────────────────────

FILE: Ecom.Catalog.Domain/Entities/Product.cs

────────────────────────────────────────────────────────────────────
namespace Ecom.Catalog.Domain.Entities;

public class Product
{
    public int Id { get; set; }
    public string Name { get; set; } = string.Empty;
    public string SKU { get; set; } = string.Empty;
    public int CategoryId { get; set; }
    public int? BrandId { get; set; }
    public int StatusId { get; set; } = 1;
    public int CreatedBy { get; set; }
    public bool IsDeleted { get; set; } = false;
    public DateTime CreatedAt { get; set; } = DateTime.UtcNow;
    public DateTime? UpdatedAt { get; set; }
    public byte[]? RowVersion { get; set; }

    public Category Category { get; set; } = null!;
    public Brand? Brand { get; set; }
    public ProductStatus Status { get; set; } = null!;
    public ICollection<ProductVariantCombination> Variants { get; set; } = new List<ProductVariantCombination>();
    public ICollection<MediaAsset> MediaAssets { get; set; } = new List<MediaAsset>();
}
────────────────────────────────────────────────────────────────────

LINE-BY-LINE EXPLANATION:
─────────────────────────

Line 1: namespace Ecom.Catalog.Domain.Entities;
WHY: Organizes code into logical groups
WHAT: All domain entities live in this namespace
BENEFIT: Clear separation from Application/Infrastructure

Line 3: public class Product
WHY: Represents the core business entity
WHAT: A product in the e-commerce catalog
REAL-WORLD: Like a product in a retail store

Line 5: public int Id { get; set; }
WHY: Primary key for database
WHAT: Unique identifier for each product
TYPE: int (auto-increment in SQL Server)
EXAMPLE: 1, 2, 3, 4...

Line 6: public string Name { get; set; } = string.Empty;
WHY: Product display name
WHAT: Human-readable product name
DEFAULT: string.Empty (prevents null reference exceptions)
EXAMPLE: "iPhone 15 Pro Max", "Nike Air Jordan"
CONSTRAINT: MaxLength(150) - defined in Infrastructure layer

Line 7: public string SKU { get; set; } = string.Empty;
WHY: Stock Keeping Unit - unique product identifier
WHAT: Business identifier (not database ID)
UNIQUE: Must be unique across all products
EXAMPLE: "IPHONE-15-PRO-256GB-BLACK"
REAL-WORLD: Like a barcode in a store
WHY SEPARATE FROM ID: 
  - ID is technical (database)
  - SKU is business (inventory management)
  - SKU can be human-readable
  - SKU can follow company naming conventions

Line 8: public int CategoryId { get; set; }
WHY: Foreign key to Category table
WHAT: Links product to a category
REQUIRED: Every product must have a category
EXAMPLE: 1 (Electronics), 2 (Clothing)

Line 9: public int? BrandId { get; set; }
WHY: Foreign key to Brand table
WHAT: Links product to a brand
NULLABLE: ? means optional (some products have no brand)
EXAMPLE: 5 (Apple), 10 (Nike), null (generic product)
WHY NULLABLE: Not all products have brands (e.g., generic items)

Line 10: public int StatusId { get; set; } = 1;
WHY: Product lifecycle status
WHAT: Current state of the product
DEFAULT: 1 (Draft status)
EXAMPLE: 1=Draft, 2=Active, 3=Discontinued
WHY NOT ENUM: Stored in database for flexibility

Line 11: public int CreatedBy { get; set; }
WHY: Audit trail - who created this product
WHAT: UserId from Identity service
EXAMPLE: 42 (user ID)
COMPLIANCE: Required for GDPR, SOX auditing

Line 12: public bool IsDeleted { get; set; } = false;
WHY: Soft delete pattern
WHAT: Marks product as deleted without removing from database
DEFAULT: false (not deleted)
WHY NOT HARD DELETE:
  - Maintains referential integrity
  - Audit trail preserved
  - Can restore if needed
  - Historical orders still reference product

Line 13: public DateTime CreatedAt { get; set; } = DateTime.UtcNow;
WHY: Timestamp of creation
WHAT: When product was created
DEFAULT: Current UTC time
WHY UTC: Consistent across time zones

Line 14: public DateTime? UpdatedAt { get; set; }
WHY: Timestamp of last update
WHAT: When product was last modified
NULLABLE: null until first update
PATTERN: Set in service layer on update

Line 15: public byte[]? RowVersion { get; set; }
WHY: Optimistic concurrency control
WHAT: EF Core concurrency token
HOW IT WORKS:
  1. User A reads Product (RowVersion = 1)
  2. User B reads Product (RowVersion = 1)
  3. User A updates → RowVersion = 2
  4. User B tries to update → CONFLICT (RowVersion mismatch)
PREVENTS: Lost updates in concurrent scenarios
REAL-WORLD: Two people editing same product simultaneously

Line 17: public Category Category { get; set; } = null!;
WHY: Navigation property for EF Core
WHAT: Related Category object
null!: Tells compiler "trust me, EF Core will populate this"
BENEFIT: Can access product.Category.Name without separate query
LAZY LOADING: Not loaded by default (use Include())

Line 18: public Brand? Brand { get; set; }
WHY: Navigation property for Brand
NULLABLE: Brand is optional
EXAMPLE: product.Brand?.Name (safe navigation)

Line 19: public ProductStatus Status { get; set; } = null!;
WHY: Navigation property for Status
WHAT: Related ProductStatus object
BENEFIT: product.Status.StatusName

Line 20: public ICollection<ProductVariantCombination> Variants { get; set; } = new List<ProductVariantCombination>();
WHY: One-to-Many relationship
WHAT: Product can have multiple variants
EXAMPLE: T-Shirt → [Small-Red, Medium-Blue, Large-Green]
INITIALIZED: new List<> prevents null reference

Line 21: public ICollection<MediaAsset> MediaAssets { get; set; } = new List<MediaAsset>();
WHY: One-to-Many relationship
WHAT: Product can have multiple images
EXAMPLE: iPhone → [front.jpg, back.jpg, side.jpg]
BENEFIT: product.MediaAssets.Where(m => m.IsPrimary)

DESIGN DECISIONS:
─────────────────

1. WHY NO METHODS IN ENTITY?
   - This is an ANEMIC DOMAIN MODEL
   - Business logic in services (Application layer)
   - Entities are just data containers
   - ALTERNATIVE: Rich Domain Model (methods like product.Publish())

2. WHY NO DATA ANNOTATIONS?
   - Clean Architecture principle
   - Domain should be framework-agnostic
   - Configurations in Infrastructure layer (Fluent API)
   - BENEFIT: Can use same entities with different ORMs

3. WHY NAVIGATION PROPERTIES?
   - EF Core convention
   - Enables eager loading (Include())
   - Simplifies queries
   - ALTERNATIVE: Just foreign keys (more manual work)

3.2 CATEGORY ENTITY - SELF-REFERENCING HIERARCHY
─────────────────────────────────────────────────

FILE: Ecom.Catalog.Domain/Entities/Category.cs

────────────────────────────────────────────────────────────────────
namespace Ecom.Catalog.Domain.Entities;

public class Category
{
    public int Id { get; set; }
    public string Name { get; set; } = string.Empty;
    public int? ParentCategoryId { get; set; }
    public bool IsDeleted { get; set; } = false;
    public DateTime CreatedAt { get; set; } = DateTime.UtcNow;
    public DateTime? UpdatedAt { get; set; }
    public byte[]? RowVersion { get; set; }

    public Category? ParentCategory { get; set; }
    public ICollection<Category> SubCategories { get; set; } = new List<Category>();
    public ICollection<Product> Products { get; set; } = new List<Product>();
}
────────────────────────────────────────────────────────────────────

KEY CONCEPT: SELF-REFERENCING RELATIONSHIP
───────────────────────────────────────────

Line 7: public int? ParentCategoryId { get; set; }
WHY: Points to parent category in same table
NULLABLE: Root categories have no parent (null)
EXAMPLE:
  - Electronics (ParentCategoryId = null) ← ROOT
  - Laptops (ParentCategoryId = 1) ← Child of Electronics
  - Gaming Laptops (ParentCategoryId = 2) ← Child of Laptops

Line 13: public Category? ParentCategory { get; set; }
WHY: Navigation to parent
EXAMPLE: gamingLaptops.ParentCategory.Name → "Laptops"

Line 14: public ICollection<Category> SubCategories { get; set; }
WHY: Navigation to children
EXAMPLE: electronics.SubCategories → [Laptops, Phones, Tablets]

REAL-WORLD HIERARCHY:
─────────────────────
Electronics (Id=1, ParentId=null)
  ├── Laptops (Id=2, ParentId=1)
  │     ├── Gaming Laptops (Id=3, ParentId=2)
  │     └── Business Laptops (Id=4, ParentId=2)
  ├── Phones (Id=5, ParentId=1)
  │     ├── Smartphones (Id=6, ParentId=5)
  │     └── Feature Phones (Id=7, ParentId=5)
  └── Tablets (Id=8, ParentId=1)

QUERY EXAMPLES:
───────────────
// Get all root categories
var roots = categories.Where(c => c.ParentCategoryId == null);

// Get breadcrumb: Gaming Laptops → Laptops → Electronics
var breadcrumb = new List<string>();
var current = gamingLaptops;
while (current != null) {
    breadcrumb.Insert(0, current.Name);
    current = current.ParentCategory;
}

WHY THIS DESIGN:
────────────────
✓ Unlimited nesting levels
✓ Easy to add new categories
✓ Simple database structure (one table)
✓ Flexible hierarchy
✗ Complex queries (recursive CTEs)
✗ Performance issues with deep nesting

──────────────────

PROBLEM WITHOUT EVENTS:
When a product is created, you need to:
1. Save to database
2. Update search index (Elasticsearch)
3. Send notification email
4. Update analytics
5. Invalidate cache

SOLUTION: Publish event, let other services subscribe

EVENT FLOW:
───────────
1. ProductService creates product
2. Publishes ProductCreatedEvent to RabbitMQ
3. Search Service subscribes → updates Elasticsearch
4. Notification Service subscribes → sends email
5. Analytics Service subscribes → updates dashboard
6. Workflow Service subscribes → updates read model

BENEFITS:
─────────
✓ Loose coupling (services don't know about each other)
✓ Scalability (add new subscribers without changing code)
✓ Resilience (if one service fails, others continue)
✓ Audit trail (event log)

4.1 DTOs (DATA TRANSFER OBJECTS)
────────────────────────────────

WHY DTOs?
─────────

PROBLEM WITHOUT DTOs:
Controller returns domain entities directly:
```csharp
[HttpGet]
public Product GetProduct(int id) => _repo.GetById(id);
```

ISSUES:
1. Exposes internal structure (RowVersion, IsDeleted)
2. Circular references (Product → Category → Products)
3. Over-fetching (client gets all data)
4. Breaking changes (change entity → break API)
5. Security (expose sensitive fields)

SOLUTION: DTOs
──────────────

FILE: Ecom.Catalog.Application/DTOs/ProductDto.cs

            .Include(p => p.Brand)
            .Include(p => p.Status)

```sql
FROM Products p
LEFT JOIN Categories c ON p.CategoryId = c.Id
LEFT JOIN Brands b ON p.BrandId = b.Id
LEFT JOIN ProductStatuses s ON p.StatusId = s.Id
WHERE p.IsDeleted = 0;
```

public class ProductConfiguration : IEntityTypeConfiguration<Product>
{
    public void Configure(EntityTypeBuilder<Product> builder)
    {
        builder.HasKey(p => p.Id);
        builder.Property(p => p.Name).IsRequired().HasMaxLength(150);
        builder.Property(p => p.SKU).IsRequired().HasMaxLength(100);
        builder.HasIndex(p => p.SKU).IsUnique();
        builder.HasIndex(p => new { p.Name, p.SKU }).HasDatabaseName("IX_Product_Search");
        builder.Property(p => p.RowVersion).IsRowVersion();

        builder.HasOne(p => p.Category).WithMany(c => c.Products).HasForeignKey(p => p.CategoryId);
        builder.HasOne(p => p.Brand).WithMany().HasForeignKey(p => p.BrandId);
        builder.HasOne(p => p.Status).WithMany().HasForeignKey(p => p.StatusId);
    }
}

public class CatalogEventPublisher
{
    private readonly ILogger<CatalogEventPublisher> _logger;
    private readonly IConfiguration _configuration;
    private IConnection? _connection;
    private IModel? _channel;

    public CatalogEventPublisher(ILogger<CatalogEventPublisher> logger, IConfiguration configuration)
    {
        _logger = logger;
        _configuration = configuration;
        InitializeRabbitMQ();
    }

    private void InitializeRabbitMQ()
    {
        try
        {
            var factory = new ConnectionFactory
            {
                HostName = _configuration["RabbitMq:Host"] ?? "localhost",
                Port = int.Parse(_configuration["RabbitMq:Port"] ?? "5672"),
                UserName = _configuration["RabbitMq:Username"] ?? "guest",
                Password = _configuration["RabbitMq:Password"] ?? "guest"
            };

            _connection = factory.CreateConnection();
            _channel = _connection.CreateModel();

            _channel.ExchangeDeclare(
                exchange: "catalog-events",
                type: ExchangeType.Topic,
                durable: true,
                autoDelete: false);

            _logger.LogInformation("RabbitMQ connection established successfully");
        }
        catch (Exception ex)
        {
            _logger.LogWarning(ex, "Failed to connect to RabbitMQ. Events will be logged only.");
        }
    }

    public Task PublishProductCreatedAsync(ProductCreatedEvent @event)
    {
        _logger.LogInformation("Publishing ProductCreatedEvent for ProductId: {ProductId}", @event.ProductId);
        
        if (_channel != null && _channel.IsOpen)
        {
            var message = JsonSerializer.Serialize(@event);
            var body = Encoding.UTF8.GetBytes(message);

            var properties = _channel.CreateBasicProperties();
            properties.Persistent = true;
            properties.ContentType = "application/json";

            _channel.BasicPublish(
                exchange: "catalog-events",
                routingKey: "product.created",
                basicProperties: properties,
                body: body);

            _logger.LogInformation("ProductCreatedEvent published to RabbitMQ");
        }
        else
        {
            _logger.LogWarning("RabbitMQ channel not available. Event logged only.");
        }

        return Task.CompletedTask;
    }
}
────────────────────────────────────────────────────────────────────

RABBITMQ CONCEPTS:
──────────────────

EXCHANGE: "catalog-events"
TYPE: Topic (routing based on patterns)
DURABLE: Survives broker restart
ROUTING KEY: "product.created"

SUBSCRIBERS CAN BIND:
- "product.*" → All product events
- "product.created" → Only creation events
- "#" → All events

MESSAGE FLOW:
─────────────
1. ProductService creates product
2. Calls CatalogEventPublisher.PublishProductCreatedAsync()
3. Event serialized to JSON
4. Published to "catalog-events" exchange
5. RabbitMQ routes to bound queues
6. Workflow Service consumes → updates read model
7. Search Service consumes → updates Elasticsearch
8. Notification Service consumes → sends email

RESILIENCE:
───────────
Line 38-40: Graceful degradation
IF RabbitMQ unavailable → Log warning, continue
WHY: Don't fail product creation if messaging is down
ALTERNATIVE: Use Outbox pattern (store events in DB, retry later)

public class ExceptionHandlingMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<ExceptionHandlingMiddleware> _logger;

    public ExceptionHandlingMiddleware(RequestDelegate next, ILogger<ExceptionHandlingMiddleware> logger)
    {
        _next = next;
        _logger = logger;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        try { await _next(context); }
        catch (KeyNotFoundException ex)
        {
            await WriteAsync(context, HttpStatusCode.NotFound, ex.Message);
        }
        catch (InvalidOperationException ex)
        {
            await WriteAsync(context, HttpStatusCode.Conflict, ex.Message);
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Unhandled exception");
            await WriteAsync(context, HttpStatusCode.InternalServerError, "An unexpected error occurred.");
        }
    }

    private static Task WriteAsync(HttpContext ctx, HttpStatusCode code, string msg)
    {
        ctx.Response.StatusCode = (int)code;
        ctx.Response.ContentType = "application/json";
        return ctx.Response.WriteAsync(JsonSerializer.Serialize(new { error = msg }));
    }
}
────────────────────────────────────────────────────────────────────

MIDDLEWARE PIPELINE:
────────────────────

Request → ExceptionHandlingMiddleware → RequestLoggingMiddleware → RateLimitingMiddleware → Authentication → Authorization → Controller → Response

EXCEPTION MAPPING:
──────────────────

KeyNotFoundException → 404 Not Found
USE CASE: Product not found
EXAMPLE: GET /api/v1/products/999 (doesn't exist)

InvalidOperationException → 409 Conflict
USE CASE: Duplicate SKU, business rule violation
EXAMPLE: POST /api/v1/products (SKU already exists)

Exception (catch-all) → 500 Internal Server Error
USE CASE: Unexpected errors
EXAMPLE: Database connection failure

WHY MIDDLEWARE?
───────────────
✓ Centralized error handling
✓ Consistent error responses
✓ No try-catch in every controller
✓ Logging in one place

DEPENDENCY INJECTION LIFETIMES:
────────────────────────────────

AddScoped:
- Created once per HTTP request
- Shared within request
- Disposed after request
USE FOR: DbContext, Repositories, Services

AddSingleton:
- Created once for application lifetime
- Shared across all requests
- Never disposed (until app shutdown)
USE FOR: Configuration, Event publishers

AddTransient:
- Created every time requested
- Not shared
- Disposed immediately
USE FOR: Lightweight, stateless services

15.3 COMMON INTERVIEW QUESTIONS & ANSWERS
──────────────────────────────────────────

Q1: Why Clean Architecture over traditional N-tier?
A: Clean Architecture inverts dependencies - the domain is at the center with zero
   dependencies. This makes business logic testable without infrastructure, allows
   swapping databases without touching domain code, and enables independent evolution
   of layers. N-tier couples layers tightly, making changes cascade.

Q2: Explain the Repository Pattern. Why use it?
A: Repository abstracts data access behind an interface. Services depend on IRepository,
   not concrete implementations. Benefits: testability (mock repositories), database
   agnosticism (swap EF Core for Dapper), centralized query logic, and follows
   Dependency Inversion Principle.

Q3: What is CQRS and why did you implement it?
A: Command Query Responsibility Segregation separates reads and writes. Write model
   (Product, Category) is normalized for data integrity. Read model (ProductReadModel)
   is denormalized for performance - no joins needed. Updated asynchronously via events.
   Benefits: optimized queries, scalability (separate read/write databases), flexibility.

Q4: How do you handle concurrency?
A: Optimistic concurrency using RowVersion. When two users update simultaneously:
   1. Both read product (RowVersion = 1)
   2. First user updates → RowVersion = 2
   3. Second user updates → DbUpdateConcurrencyException (RowVersion mismatch)
   4. Middleware returns 409 Conflict
   Alternative: Pessimistic locking (row locks) - not used due to performance impact.

Q5: Explain your event-driven architecture.
A: When products are created/updated, domain events are published to RabbitMQ.
   Other services subscribe: Workflow updates read model, Search updates Elasticsearch,
   Notification sends emails. Benefits: loose coupling, scalability, resilience.
   If one service fails, others continue. Uses Topic exchange for flexible routing.

Q6: How do you ensure data consistency across services?
A: Eventual consistency model. Write to database first, then publish event.
   For stronger guarantees, implement Outbox pattern: store events in database,
   separate process publishes to message queue. This ensures events are never lost.

Q7: Why soft delete instead of hard delete?
A: Soft delete (IsDeleted flag) maintains referential integrity - historical orders
   still reference products. Enables data recovery, preserves audit trail, and
   prevents cascading deletes. Trade-off: database grows larger, queries must filter.

Q8: How do you handle authentication and authorization?
A: JWT bearer tokens for authentication. Token contains claims (UserId, Email, Roles).
   Role-based authorization using [Authorize(Roles = "Admin")] attributes.
   Middleware validates token signature, expiration, issuer, and audience.
   For service-to-service, would add API key authentication.

Q9: Explain your testing strategy.
A: Unit tests mock dependencies (repositories, services) to test business logic in
   isolation. Fast, no database needed. Integration tests use WebApplicationFactory
   with in-memory database to test full request/response cycle, including middleware,
   authentication, and database interactions. Aim for 80%+ code coverage.

Q10: How would you scale this service?
A: Horizontal scaling: stateless API, multiple instances behind load balancer.
   Database scaling: read replicas for queries (CQRS helps), write to master.
   Caching: Redis for ProductReadModel. Message queue: RabbitMQ cluster.
   Monitoring: distributed tracing (OpenTelemetry), metrics (Prometheus),
   centralized logging (ELK). Circuit breaker pattern for external dependencies.

Q11: What are the main challenges you faced?
A: 1. Circular references in navigation properties - solved with DTOs
   2. N+1 query problem - solved with Include() eager loading
   3. Concurrent updates - solved with optimistic concurrency
   4. Event delivery guarantees - would implement Outbox pattern
   5. Testing with external dependencies - solved with mocking

Q12: How do you handle errors?
A: Centralized exception handling middleware catches all exceptions.
   Maps to HTTP status codes: KeyNotFoundException → 404, InvalidOperationException → 409.
   Structured logging with correlation IDs for tracing. Never expose stack traces
   to clients (security). Log errors with context for debugging.

Q13: Explain your database design decisions.
A: Code-First EF Core with Fluent API configurations. Self-referencing Category
   table for hierarchy. Composite indexes for search performance. RowVersion for
   concurrency. Soft delete for data recovery. Separate ProductReadModel for CQRS.
   Audit log table for compliance. Normalized write model, denormalized read model.

Q14: What would you do differently?
A: 1. Implement Outbox pattern for guaranteed event delivery
   2. Add Redis caching for read model
   3. Implement API Gateway for centralized authentication
   4. Add distributed tracing (OpenTelemetry)
   5. Implement circuit breaker for external services
   6. Add comprehensive integration tests
   7. Implement blue-green deployment strategy

Q15: How do you ensure code quality?
A: 1. SOLID principles throughout
   2. Clean Architecture for separation of concerns
   3. Unit tests for business logic
   4. Integration tests for API endpoints
   5. Code reviews (would use pull requests)
   6. Static analysis (SonarQube)
   7. Consistent naming conventions
   8. Comprehensive logging
   9. Documentation (XML comments, README)
